# ADR-001: Fundación multi-tenant + RBAC + actor canónico para Urbania SaaS

> Decisión tomada en 2026-06-28. Estado: Accepted.

## Principio rector

Separar dos planos y no tocar Auth existente:

1. **Identidad + tenant** (quién eres, de qué cuenta) → extiende lo que ya hay agregando `organization_id` a las tablas existentes sin reescribir lógica de login/MFA/tokens.
2. **Autorización** (qué puedes hacer y dónde) → capa nueva `src/Authorization` que se monta **encima** del Auth existente. La capa nueva envuelve, no reemplaza.

Auth (identidad, login, MFA, tokens — 259 tests) no se reescribe. Se le añade el tenant y se le monta encima un módulo de autorización.

---

## Contexto

El sistema Urbania comenzó como un MVP con autenticación binaria (admin/user) y un modelo plano de usuarios. Tras implementar Propiedades (con `condominium_id` como raíz de scoping) y Directorio (con `contacts` separado de `users`), se identificaron 4 hallazgos en la auditoría de integridad ([[_AUDITORIA_INTEGRIDAD_2026-06-28]]) que comparten el mismo sustrato:

- **C2 — No existe capa multi-tenant**: `users` sin `organization_id`, `condominiums` es raíz sin entidad propietaria. Cada tabla operativa nueva necesitará scoping.
- **C3 — Roles binarios (admin/user) incompatibles con los ~14 roles planeados**: no hay tabla de roles/permisos, el JWT lleva el claim `role` directamente de `users.role`.
- **H1 — `users.unit` (texto libre) + actor de operaciones sin definir**: conviven dos formas de asociar persona↔unidad, y no está decidido si las tablas operativas usan `user_id` o `contact_id`.
- **H2 — Staff sin scoping a conjunto**: un admin/staff no tiene relación formal con el/los condominio(s) donde opera.

El modelo de negocio (SaaS multi-tenant con administradoras que gestionan múltiples conjuntos) exige resolver estos cuatro puntos de forma coherente antes de escalar a las features 5–16.

---

## Decisión

Se adoptan las siguientes decisiones de arquitectura, que se implementarán en el orden descrito dentro de CAMBIO-006:

### 1. Multi-tenancy — base compartida + organization_id discriminador + RLS

**Modelo**: `organizations` (la cuenta) 1—N `condominiums` (el conjunto). Un edificio único = una org con un conjunto, sin casos especiales.

**Implementación**:
- Tabla `organizations` como raíz tenant.
- Agregar `organization_id` a `users`, `condominiums` y catálogos de nivel org.
- Lo operativo se scopea por `condominium_id` (que ya implica la org), reusando el patrón existente en `properties.condominium_id`.
- Para aislamiento: **PostgreSQL Row-Level Security (RLS)** con `current_setting('app.org_id')` fijado por request desde el JWT en un middleware de `Shared/Infrastructure/Middleware`.
- **Fases**: arrancar con **global scopes de Eloquent** + la columna (bajo riesgo, inmediato). Activar RLS como **fast-follow** (la columna lo hace trivial). RLS es la garantía a nivel BD de que un `WHERE` olvidado no filtre datos entre tenants.
- **No** a schema-por-tenant: el perfil es "muchos tenants pequeños". Base compartida escala a miles y abarata migraciones. Si un cliente enterprise exige aislamiento físico, se migra ese tenant individualmente.

### 2. RBAC — roles/permissions/role_assignments con scope, permisos resueltos server-side

**Modelo**:
- Permisos como catálogo fijo `recurso.accion` sembrado por el operador (el usuario edita roles, no el vocabulario).
- Roles de sistema (~14 roles planeados) + roles personalizados por org.
- `role_assignments(user_id × role_id × scope_type × scope_id × vigencia)` como tabla central.

**Resolución de permisos**:
- El JWT **no** lleva los permisos (bloat + se quedan viejos). Lleva solo identidad + `org_id`.
- Los permisos efectivos se resuelven **por request** contra las tablas, cacheados en Redis (ya usado en el sistema) con TTL corto o invalidación al cambiar un rol.
- Un `Gate` en `Shared` expone `can('pagos.aprobar', $scope)`.
- Módulo nuevo `src/Authorization` separado de `Auth`.
- **No** a ABAC: `recurso.acción` + scope + umbrales de aprobación cubre el requerimiento sin sobre-ingeniería.

**Impacto en JWT**:
- El claim `role` del JWT se elimina (o se deja como metadata informativa no vinculante). La autorización real se resuelve server-side.

### 3. Actor canónico — "party = contact, actor = user"

Se distinguen dos preguntas:
- **Pertenencia** (dueño de la cuenta de cobro, residente de la reserva, radicante de la PQRS) → `contact_id` (+ `unit_id`), porque no toda persona tiene login (propietario ausente registrado por Ley 675).
- **Autoría** (created_by, quién registró el pago, quién aprobó) → `user_id`, porque la acción la ejecuta alguien autenticado.

**Invariante**: todo `user` tiene su `contact` (`contacts.user_id`), así staff y residentes son "personas" de forma uniforme y el actor siempre mapea a una party.

**Deprecación de `users.unit`**:
- Backfill: crear el `contact` faltante por usuario, migrar el texto de `unit` a `property_occupants` donde haga match, reconciliar el resto manualmente (es volumen bajo — datos de la era Auth).
- Eliminar la columna `users.unit`.
- `property_occupants` queda como única verdad de persona↔unidad.

### 4. Scoping de staff — via role_assignments.scope, sin tabla aparte

- **No** crear `user_condominiums`. El alcance vive en `role_assignments(scope_type ∈ {organization, condominium, tower, unit})`.
- Admin de administradora → rol @ `organization`.
- Administrador de conjunto → rol @ `condominium X`.
- Vigilante → rol @ `condominium X` (o tower si aplica).
- Para residentes: su alcance se deriva de `property_occupants` (sus unidades → conjunto) + un set fijo "Residente".
- El resolver mezcla las dos fuentes (asignaciones explícitas para staff, ocupación para residentes) y ahorra sincronizar miles de filas.

## Secuencia de implementación (dentro de CAMBIO-006)

1. **C2** — `organizations` + `organization_id` + contexto de tenant en Shared (global scopes; RLS fast-follow).
2. **H1** — Invariante `user⇄contact`, regla party/actor, deprecar `users.unit`.
3. **C3 + H2** juntos — Tablas RBAC con scope; el role del JWT pasa a resolución server-side; añadir `org_id` al token.

**Cuidado con el orden de backfill**: la org "por defecto" debe existir antes de poblar `organization_id`; los contacts antes de migrar `users.unit`. Conviene un seeder de bootstrap que cree la organización del primer tenant y le cuelgue el conjunto y usuarios actuales.

## Alternativas consideradas

| Alternativa | Por qué se descartó |
|---|---|
| **Schema-por-tenant** (DB separada por cliente) | No escala para "muchos tenants pequeños". Migraciones costosas, operación compleja. Se reserva para casos enterprise que exijan aislamiento físico. |
| **Permisos en JWT** (incluir lista de permisos en el access token) | Bloat del token (14 roles × recursos = decenas de claims). Los permisos se quedan viejos hasta que expire el token. Inseguro si no se revocan inmediatamente. |
| **ABAC** (Attribute-Based Access Control) | Sobredimensionado para el perfil actual. `recurso.acción` + scope + umbrales de aprobación cubre todas las reglas de negocio identificadas sin la complejidad de ABAC. |
| **Tabla `user_condominiums`** para scoping de staff | Duplicación con `role_assignments.scope`. Una tabla única de asignaciones con scope_type resuelve staff y residentes sin sincronización cruzada. |
| **`user_id` como actor único** en tablas operativas | Excluye a propietarios sin login (Ley 675 obliga a registrarlos). El modelo dual `contact_id` (pertenencia) + `user_id` (autoría) cubre ambos casos. |

---

## Consecuencias

**Positivas:**
- Auth existente (259 tests) queda intocable — bajo riesgo.
- La columna `organization_id` ya existente como patrón en `condominium_id` — extensión natural.
- RBAC resuelto server-side evita el problema de permisos stale en JWT.
- RLS como fast-follow da garantía BD de aislamiento sin reescribir queries.
- El modelo dual `contact`/`user` cumple con Ley 675 (libro de propietarios).
- Los scope_type polimórficos (`organization/condominium/tower/unit`) cubren todos los niveles granulares identificados.

**Negativas / trade-offs:**
- Requiere migración de datos: backfill de `organization_id`, creación de contacts desde users existentes, deprecación de `users.unit`.
- El RBAC server-side agrega latencia de Redis (mínima, pero existe).
- El seeder de bootstrap del primer tenant debe existir antes de migrar cualquier tabla.

**Neutrales / notas:**
- El orden de implementación importa: C2 → H1 → C3+H2. No se puede saltar pasos.
- Los catálogos de nivel org (property_types, property_statuses, occupant_types, property_document_types) también reciben `organization_id`.
- El middleware de tenant debe estar en todos los requests autenticados desde el primer deploy.

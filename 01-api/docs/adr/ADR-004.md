# ADR-004: Doble token (Access + Refresh) con rotación

## Contexto

El mecanismo de autenticación del API necesita balancear seguridad y experiencia de usuario. Un token único de larga duración es inseguro (si se filtra, el atacante tiene acceso prolongado). Un token de corta duración obliga al usuario a autenticarse constantemente, degradando la experiencia.

Los clientes Web (SPA) y App (móvil) necesitan mantener sesiones activas sin pedir credenciales repetidamente, pero sin exponer tokens de larga duración en el cliente.

## Decisión

Implementar una **estrategia de doble token con rotación**:

| Aspecto | Access Token | Refresh Token |
|---|---|---|
| Formato | JWT firmado RS256 | Opaco (hash SHA-256 almacenado en BD) |
| Duración | 15 minutos | 7 días (renovable) |
| Almacenamiento en cliente | Memoria (Zustand / variable en memoria) | Memoria (NUNCA localStorage) |
| Rotación | No aplica (nuevo en cada refresh) | Cada uso genera un nuevo token; el anterior se invalida |
| Detección de robo | — | Si un token ya rotado se reusa, se invalida toda la familia (`token_family`) |

### Flujo de rotación

1. El cliente tiene access_token (15 min) y refresh_token (7 días).
2. Al expirar el access_token, el cliente envía el refresh_token a `POST /auth/refresh`.
3. El API valida el hash, emite un **nuevo par** (nuevo access_token + nuevo refresh_token), e invalida el refresh_token anterior.
4. Si alguien reusa un refresh_token ya rotado, se detecta como **rotación ilegítima** y se invalida toda la familia (`token_family`), forzando re-autenticación completa.

Límite de seguridad: máximo 50 rotaciones por familia de tokens. Superado ese límite, se registra un evento `suspicious_activity`.

## Alternativas consideradas

| Alternativa | Razón de descarte |
|---|---|
| Access token de larga duración (7 días) | Inseguro: si se filtra, el atacante tiene acceso por 7 días. Sin posibilidad de revocación sin blacklist. |
| Solo refresh token (sin access token) | El refresh token se enviaría en cada request, aumentando la superficie de ataque. El access token JWT permite validación stateless. |
| Cookies httpOnly automáticas | No es viable para App móvil. La estrategia de token en memoria con Bearer header funciona igual para ambos clientes. |
| Sin rotación de refresh token | Un refresh token robado es válido por 7 días sin posibilidad de detección. |

## Consecuencias

**Positivas:**
- Access token de corta duración minimiza la ventana de exposición.
- Detección de robo de refresh token vía rotación ilegítima.
- Sin almacenamiento persistente de tokens en el cliente (resistente a XSS).
- Familia de tokens permite rastrear toda la cadena de rotación.

**Negativas:**
- Mayor complejidad en el cliente (interceptores HTTP, cola de requests durante refresh).
- Costo de BD/Redis por cada refresh (validar hash, rotar, invalidar anterior).
- Ventana de 15 minutos donde un access token robado es válido (mitigado vía blacklist JTI en Redis).

# ADR-001: Clean Architecture + DDD sobre MVC Laravel estándar

> Crear en `docs/adr/ADR-001.md`. Cada ADR justifica una decisión técnica fundamental ya aplicada en el código (ver [[API_ARCHITECTURE]] §14), no una decisión teórica.

## Contexto

El proyecto Urbania API necesita soportar múltiples módulos de negocio (Auth, Propiedades, Residentes, Cuotas, Pagos, Mora, etc.) que crecerán en complejidad con el tiempo. Laravel por defecto promueve una arquitectura MVC donde la lógica de negocio tiende a acumularse en controladores o modelos Eloquent, generando código difícil de testear y mantener a medida que el dominio crece.

Se evaluó si la estructura MVC estándar de Laravel era suficiente para el tamaño y complejidad esperados del proyecto.

## Decisión

Adoptar **Clean Architecture + Domain-Driven Design (DDD)** con separación estricta en 4 capas:

- **Domain**: Entidades, Value Objects, Excepciones, Eventos, Interfaces de repositorio — cero dependencias externas.
- **Application**: Casos de uso, DTOs, servicios de aplicación — orquesta el dominio.
- **Infrastructure**: Implementaciones concretas (Eloquent, JWT, HTTP), mappers, controladores.
- **Presentation**: Rutas, Service Providers, middleware — configuración del framework.

Cada módulo de negocio vive en un **bounded context** independiente dentro de `src/`, comunicándose solo vía `Shared/` o eventos de dominio.

## Alternativas consideradas

| Alternativa | Razón de descarte |
|---|---|
| MVC Laravel estándar | La lógica de negocio se mezcla con infraestructura, difícil de testear y mantener a escala. Los modelos Eloquent se convierten en "God Objects". |
| Arquitectura hexagonal pura | Similar en espíritu, pero Clean Architecture es más conocida por el equipo y tiene más ejemplos de referencia en el ecosistema PHP/Laravel. |
| Microservicios | Sobredimensionado para el alcance actual; el monolito modular con bounded contexts permite separación sin la sobrecarga operativa de microservicios. |

## Consecuencias

**Positivas:**
- El Domain es puramente PHP sin dependencias de Laravel, testeable de forma aislada (99.25% cobertura).
- Los bounded contexts evitan el acoplamiento entre módulos.
- Cambiar de infraestructura (BD, JWT provider) no afecta la lógica de negocio.
- Architecture tests verifican automáticamente las reglas de dependencia.

**Negativas:**
- Más archivos y boilerplate que MVC estándar para módulos simples.
- Curva de aprendizaje inicial para desarrolladores acostumbrados a MVC de Laravel.

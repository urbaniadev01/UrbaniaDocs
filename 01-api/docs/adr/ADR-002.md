# ADR-002: RS256 sobre HS256 para firma JWT

## Contexto

El API de Urbania necesita autenticación stateless mediante tokens JWT. Laravel no incluye un mecanismo nativo de JWT, por lo que se requiere un paquete externo (`php-open-source-saver/jwt-auth`) y una decisión sobre el algoritmo de firma.

HS256 (HMAC-SHA256) es el algoritmo por defecto en la mayoría de paquetes JWT y es más simple de configurar (un único secreto compartido). Sin embargo, en un escenario donde múltiples servicios cliente (Web, App) consumen el API y potencialmente servicios internos necesitan validar tokens, compartir un secreto simétrico incrementa la superficie de ataque.

## Decisión

Usar **RS256 (RSA Signature with SHA-256)** como algoritmo exclusivo de firma JWT.

- Se genera un par de claves RSA de 4096 bits.
- La clave privada firma los tokens (solo el API).
- La clave pública se distribuye a los consumidores que necesiten validación offline.
- Rotación automática de claves cada 90 días, manteniendo 2 generaciones previas para tokens aún vigentes.

**Prohibido explícitamente:** HS256 en producción.

## Alternativas consideradas

| Alternativa | Razón de descarte |
|---|---|
| HS256 (HMAC simétrico) | Un único secreto compartido entre API y todos los clientes. Si cualquier cliente se ve comprometido, el secreto se expone y permite forjar tokens. No escalable cuando hay múltiples consumidores. |
| ES256 (ECDSA) | Curva elíptica, claves más cortas con seguridad equivalente. Sin embargo, el paquete `jwt-auth` tiene soporte más maduro y testeado para RS256. Reevaluar cuando el ecosistema madure. |
| EdDSA (Ed25519) | Aún sin soporte generalizado en paquetes JWT de PHP. |

## Consecuencias

**Positivas:**
- La clave privada nunca abandona el servidor del API.
- Los clientes pueden validar la firma del token sin conocer la clave privada.
- Rotación de claves posible sin invalidar todos los tokens simultáneamente.

**Negativas:**
- Mayor costo computacional para firmar/verificar (RSA 4096 vs HMAC).
- Complejidad adicional en la gestión del ciclo de vida de claves.
- Los clientes deben obtener y cachear la clave pública.
